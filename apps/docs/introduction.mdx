**betterhook** is a git hooks manager for teams running coding agents (Claude Code, Cursor, Codex, Aider, ...) in parallel, each in its own git worktree.

A single static Rust binary, ~6 MB stripped:

- ~30 ms cold start, ~50 ms no-op hook run
- Line-streaming subprocess I/O via Tokio — output is live, memory stays constant
- One wrapper in the shared `.git/hooks/` dir resolves the current worktree at commit time and loads *its* config
- Opt-in coordinator daemon (`betterhook serve`) that serializes tool conflicts across worktrees
- NDJSON output, machine-readable status, stable exit codes, and a `fix` subcommand for agent self-correction

```sh
betterhook init           # scaffold betterhook.toml
betterhook install        # install worktree-aware wrappers
git commit -am "go"       # your hooks run, correctly, per-worktree
```

## The problem

Several agents commit on overlapping branches, each in its own worktree, each running formatters and linters in pre-commit. They share a common git dir, a single `cargo target/`, an ESLint cache, a prettier lockfile. Under that load, hook managers built for one developer in one checkout fall over: installs fail in linked worktrees, subprocess output buffers in memory, parallel scheduling ignores priority, untracked files trip formatters.

[Why betterhook](/why-betterhook) walks through the specific lefthook issues this is built to fix.
