[Conductor](https://conductor.build) is a macOS app that spawns multiple AI
coding agents in parallel, each in its own git worktree. betterhook is
designed to plug into this workflow directly.

## What Conductor provides

- **One worktree per agent**, all rooted at the same git common dir.
- **Long-running agent processes** that run build/lint/test commands many times.
- **A shared environment** where agents can see each other's worktrees but shouldn't corrupt shared state.

## Setup

Once per repo:

```sh
betterhook init
betterhook install
```

That's it. All N worktrees that Conductor subsequently creates will resolve
through the single shared wrapper, each to their own config.

## Pre-warming the daemon

If you know every agent in a repo is going to acquire the same set of
coordinator locks, you can pre-warm `betterhookd` once at the start of a
Conductor session:

```sh
BETTERHOOK_DAEMON_SOCK=/path/to/shared.sock \
  betterhookd --socket /path/to/shared.sock &
```

Then export `BETTERHOOK_DAEMON_SOCK=/path/to/shared.sock` in each agent's
environment. The client skips auto-discovery and connects directly.

<Note>
  This is optional. Without it, the first job that declares `isolate` spawns
  the daemon on demand and subsequent agents in the same repo reuse it via
  the auto-discovery path.
</Note>

## Disabling locks per-session

Sometimes you want to run one agent in isolation (e.g. for debugging) and
bypass the coordinator entirely:

```sh
BETTERHOOK_NO_LOCKS=1 your-agent-cli
```

Every job that would normally acquire a lock runs unlocked with a single
warning on stderr.

## Monitoring

`betterhook status` is the idiomatic way for Conductor to introspect a
worktree before firing off work:

```sh
betterhook status --worktree /path/to/wt-auth | jq '.installed.hooks'
```

An agent can verify betterhook is installed, which hooks are wired up,
and whether the wrapper SHAs still match — all before touching any files.

If Conductor (or a similar harness) needs a first-class integration point
