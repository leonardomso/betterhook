---
title: "Monorepo support"
description: "Per-package hook blocks with path filters and inheritance."
icon: "boxes"
---

Betterhook v1 introduces `[packages.<name>]` blocks for monorepos. The
runner groups staged files by longest-path-prefix match against each
package's `path`, runs each matched package's hook block in parallel,
and serializes git-index writes across packages with a shared mutex.

## Declaring packages

```toml
[packages.frontend]
path = "apps/web"

[packages.api]
path = "services/api"
```

A staged file at `apps/web/src/Button.tsx` belongs to `frontend`. A
file at `services/api/src/main.rs` belongs to `api`. Files outside any
declared package fall back to the root `[hooks.*]` blocks.

## Per-package hooks

Each package can override or extend the root hook blocks:

```toml
[hooks.pre-commit.jobs.lint]
run = "echo 'default lint'"

[packages.frontend.hooks.pre-commit.jobs.lint]
run = "pnpm -F frontend lint"
reads = ["apps/web/**"]
concurrent_safe = true

[packages.frontend.hooks.pre-commit.jobs.test]
run = "pnpm -F frontend test"
```

The package's hook block inherits from the root and overlays its own
jobs:

- adding a job (`test`) extends the merged map
- overriding a job (`lint`) replaces the root's command for that
  package only
- setting `skip = true` on a job at the package level removes it

## Inspection

`betterhook status` groups hooks by package; `betterhook explain
--package frontend` scopes the output. `betterhook run pre-commit
--package frontend` runs only the matching block.

## Performance

Packages run in parallel under a top-level `Semaphore(parallel_limit)`.
Stage-fixed git-index writes are serialized through a shared async
mutex so two packages racing the same file never produce a half-staged
state.
