betterhook is a Rust cargo workspace split into two apps and an `xtask`:

```text
betterhook/
├── apps/
│   ├── betterhook/    # library crate (config, runner, cache, daemon, builtins)
│   ├── cli/           # the `betterhook` CLI (thin clap frontend)
│   └── docs/          # this site (Mintlify)
├── xtask/             # bench + stress + fuzz harness
└── packaging/         # Homebrew formula + npm wrapper scaffolds
```

## How a commit flows

<Steps>
  <Step title="You run git commit">
    ```sh
    git commit -m "feat: my change"
    ```
  </Step>
  <Step title="Git executes the pre-commit hook">
    Looks at `core.hooksPath` (default: `<git-common-dir>/hooks/`) and executes
    the `pre-commit` script. That script is **our wrapper**, written by
    `betterhook install`.
  </Step>
  <Step title="The wrapper resolves the current worktree">
    ```sh
    worktree_root="$(git rev-parse --show-toplevel)"
    ```
    This returns the current worktree root, even though the wrapper itself
    lives in the shared common dir. This is the git feature lefthook fails
    to use correctly.
  </Step>
  <Step title="The wrapper execs into betterhook">
    ```sh
    exec "/path/to/betterhook" __dispatch \
      --hook pre-commit \
      --worktree "$worktree_root" \
      --git-dir "$GIT_DIR" \
      -- "$@"
    ```
    `exec` means no extra process in the chain.
  </Step>
  <Step title="betterhook loads the worktree's config">
    Walks `$worktree_root` for `betterhook.{toml,yml,yaml,json}`, resolves
    `extends`, applies `betterhook.local.*`, lowers to the typed `Config`.
    On soft-miss (no config, no hook, no jobs) it exits 0 silently.
  </Step>
  <Step title="The runner spawns jobs">
    If `hook.parallel` is true, via a `tokio::sync::Semaphore`-bounded
    JoinSet in priority order; otherwise sequentially. Each job's subprocess
    streams output line-by-line through the central
    `OutputMultiplexer` actor.
  </Step>
  <Step title="Isolation, stash, timeouts, stage_fixed">
    Every cross-cutting feature (coordinator locks, untracked-stash safety,
    per-job timeouts, `stage_fixed` re-adding) is applied inline as the
    runner walks the job list.
  </Step>
  <Step title="Summary and exit">
    A final `summary` event fires (to TTY or NDJSON), `run_hook` returns an
    `ExecutionReport`, and the CLI maps it to the documented exit code.
    Non-zero blocks the commit.
  </Step>
</Steps>

## Design constraints

1. Worktree-native. Every git operation is scoped to an explicit worktree path; nothing relies on the ambient `$GIT_DIR`.
2. Streaming, not buffered. Tokio + `AsyncBufReadExt::lines()` keeps memory O(1) regardless of subprocess output volume.
3. Agent-first. NDJSON output, stable exit codes, machine-readable `status` and `explain`.
4. Opt-in complexity. The coordinator daemon, fuzzing, benches, and compat suite are separable. `init + install + run` is enough for a working hook.
5. Subprocess git, not libgit2/gix. We shell out to the user's `git` binary; both libgit2 and early gix conflated `$GIT_DIR` with `$GIT_COMMON_DIR`, which is the source of lefthook's worktree install bugs.
