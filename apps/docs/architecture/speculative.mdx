---
title: "Speculative execution"
description: "Prewarming concurrent_safe jobs into the cache before you commit."
icon: "zap"
---

The coordinator daemon watches every active worktree with `notify`,
debounces save events per file, resolves which `concurrent_safe` jobs
match the touched paths, and runs them through the same DAG executor
the commit-time runner uses. Results land in the CA cache. By the
time you `git commit`, the runner reads from the cache.

## Configuration

```toml
[speculative]
enabled = true
debounce_ms = 250
watch_exclude = ["**/target/**", "**/node_modules/**", "**/.git/**"]
```

## Graceful degradation

`notify` is flaky on NFS, sandboxed containers, and fresh Linux
boxes that haven't bumped their inotify watch limit. betterhook never
crashes the daemon when a watcher fails to start. It sets a
`disabled_reason` on the speculative stats sidecar and falls back to
non-speculative mode for that worktree. `betterhook status` and
`betterhook doctor` both surface the reason.

## Inspecting the runner

```bash
$ betterhook status | jq '.speculative'
{
  "watched_worktrees": 3,
  "watch_count": 12,
  "queue_depth": 0,
  "last_prewarm_ms_ago": 142,
  "disabled_reason": null
}
```

The daemon writes this snapshot to
`<common-dir>/betterhook/speculative-stats.json` after every event,
so the CLI never has to round-trip a socket query.

## When it doesn't help

- Jobs that aren't `concurrent_safe` are never speculatively run.
- Jobs whose `reads` glob doesn't match the saved file are skipped
  the same way a commit-time match would skip them.
- Jobs that mutate the file (`writes` is non-empty) are excluded —
  the cache can't faithfully replay them.
